================================================================================
COMPREHENSIVE TERMINAL EMULATOR FEATURE LIST
Research Document -- March 2026
================================================================================

Terminal Emulators Referenced:
  WT   = Windows Terminal (Microsoft)
  Kit  = Kitty (Kovid Goyal)
  Wez  = WezTerm (Wez Furlong)
  Ala  = Alacritty
  iT2  = iTerm2 (macOS)
  Hyp  = Hyper (Electron-based)
  Con  = Contour
  Foot = foot (Wayland)
  Gho  = Ghostty (Mitchell Hashimoto)
  Rio  = Rio terminal

Complexity Ratings:
  [Low]    = Straightforward; well-defined spec, small surface area
  [Medium] = Moderate; requires state management, some edge cases
  [High]   = Complex; significant architecture, cross-cutting concerns
  [V.High] = Very high; requires deep OS integration, rendering pipelines,
             or protocol-level work


================================================================================
1. CORE TERMINAL FEATURES
================================================================================

1.1  TAB MANAGEMENT
--------------------------------------------------------------------------------

1.1.1  Create New Tab
  What: Open a new terminal tab, optionally with a specific profile/shell.
  Who:  WT, Kit, Wez, iT2, Hyp, Con, Gho, Rio
  Note: Alacritty deliberately omits tabs (philosophy: use a WM or tmux).
  Complexity: [Low] -- Spawn a PTY process, attach to a tab container widget.

1.1.2  Close Tab
  What: Terminate the shell process and remove the tab.
  Who:  WT, Kit, Wez, iT2, Hyp, Con, Gho, Rio
  Complexity: [Low] -- Send SIGHUP/close handle, remove UI element.

1.1.3  Reorder Tabs (Drag and Drop)
  What: Rearrange tab order by dragging tabs in the tab bar.
  Who:  WT, Kit, Wez, iT2, Con, Gho
  Complexity: [Medium] -- Requires drag-and-drop hit testing and tab bar
  reflow logic.

1.1.4  Rename Tab
  What: Assign a custom title to a tab interactively or via escape sequence
        (OSC 0 / OSC 2).
  Who:  WT, Kit, Wez, iT2, Con, Gho
  Complexity: [Low] -- In-place text editing on the tab title widget plus
  handling OSC sequences from the shell.

1.1.5  Pin Tab
  What: Mark a tab as pinned so it stays at the left of the tab bar and
        cannot be accidentally closed.
  Who:  iT2 (via arrangements), WT (partial, via settings workaround)
  Note: Not widely implemented. Most terminals rely on protected/read-only
        pane modes instead.
  Complexity: [Low] -- UI flag on the tab model; filter close actions.

1.1.6  Duplicate Tab
  What: Clone the current tab, preserving profile, working directory, and
        environment variables.
  Who:  WT, Kit, Wez, iT2, Con, Gho
  Complexity: [Medium] -- Requires capturing CWD (via OSC 7 or /proc),
  profile settings, and environment.

1.1.7  Tab Tear-off / Attach
  What: Drag a tab out of a window to create a new window, or drag into an
        existing window to attach it.
  Who:  WT, iT2, Gho (partial)
  Complexity: [High] -- Requires cross-window drag-and-drop, PTY handle
  transfer, and window lifecycle management.

1.1.8  Tab Color
  What: Assign a custom color indicator to a tab for visual differentiation.
  Who:  WT, iT2, Wez
  Complexity: [Low] -- Color property on the tab model; render in tab bar.

1.1.9  Tab Switcher (MRU / In-Order)
  What: Overlay UI listing all open tabs for quick keyboard navigation,
        optionally sorted by most-recently-used order.
  Who:  WT (MRU and in-order modes), iT2, Wez
  Complexity: [Medium] -- Requires tracking tab activation history and
  rendering an overlay list.

1.1.10 Move Tab to New Window
  What: Action to extract the current tab into its own standalone window.
  Who:  WT, iT2
  Complexity: [Medium] -- Similar to tear-off but triggered by keybinding.


1.2  SPLIT PANES
--------------------------------------------------------------------------------

1.2.1  Horizontal Split
  What: Divide the current pane into top and bottom regions.
  Who:  WT, Kit, Wez, iT2, Con, Gho, Rio
  Complexity: [Medium] -- Binary tree or linked-list pane layout; PTY resize
  propagation.

1.2.2  Vertical Split
  What: Divide the current pane into left and right regions.
  Who:  WT, Kit, Wez, iT2, Con, Gho, Rio
  Complexity: [Medium] -- Same as horizontal but orthogonal axis.

1.2.3  Auto Split
  What: Automatically choose horizontal or vertical based on which dimension
        is larger (most-square algorithm).
  Who:  WT, iT2
  Complexity: [Low] -- Compare width vs height of current pane.

1.2.4  Directional Split (up/down/left/right)
  What: Create a new pane in a specific cardinal direction relative to the
        focused pane.
  Who:  WT, Kit (layouts), Wez
  Complexity: [Medium] -- Requires pane tree traversal to find insertion point.

1.2.5  Resize Panes
  What: Adjust pane borders with keyboard (arrow keys + modifier) or mouse
        drag.
  Who:  WT, Kit, Wez, iT2, Con, Gho
  Complexity: [Medium] -- Constraint propagation across the pane tree; minimum
  size enforcement.

1.2.6  Navigate Between Panes (Focus Movement)
  What: Move keyboard focus directionally (up/down/left/right) or by
        previous/next order.
  Who:  WT, Kit, Wez, iT2, Con, Gho
  Complexity: [Medium] -- Spatial lookup in pane tree to find neighbor in
  given direction.

1.2.7  Swap Panes
  What: Exchange the positions of two panes in the layout.
  Who:  WT, Kit, Wez
  Complexity: [Medium] -- Swap nodes in the pane tree; re-render layout.

1.2.8  Move Pane to Another Tab
  What: Detach a pane from the current tab and attach it to a different tab.
  Who:  WT, Wez
  Complexity: [High] -- Requires reparenting pane node across tab containers.

1.2.9  Zoom / Maximize Pane (Toggle)
  What: Temporarily expand the focused pane to fill the entire tab area;
        toggle back to restore original layout.
  Who:  WT, Kit, Wez, iT2, Con, Gho
  Complexity: [Medium] -- Save layout state, render single pane full-size,
  restore on toggle.

1.2.10 Toggle Split Orientation
  What: Switch an existing split from horizontal to vertical or vice versa.
  Who:  WT, Kit
  Complexity: [Low] -- Flip the split direction flag on the parent node.

1.2.11 Focus Follows Mouse (Panes)
  What: Automatically focus the pane under the mouse cursor without clicking.
  Who:  WT, Kit, Wez, iT2
  Complexity: [Low] -- Mouse-enter event handler on pane regions.

1.2.12 Pane Border Color / Indicators
  What: Highlight the focused pane border with the system accent color or a
        configured color.
  Who:  WT, Kit, Wez, Con
  Complexity: [Low] -- Conditional border rendering on focus change.


1.3  SCROLLBACK BUFFER
--------------------------------------------------------------------------------

1.3.1  Configurable Scrollback Size
  What: Set the number of lines retained in the scrollback buffer
        (historySize).
  Who:  WT (default 9001), Kit (default 2000), Wez (default 3500), Ala
        (default 10000), iT2 (configurable), Con, Foot, Gho
  Complexity: [Low] -- Ring buffer allocation; configurable capacity.

1.3.2  Infinite / Unlimited Scrollback
  What: Keep all output without any line limit, using disk-backed storage
        if needed.
  Who:  iT2 (unlimited scrollback option), Kit (scrollback_pager),
        Wez (scrollback_lines = very large)
  Note: True infinite scroll requires paging to disk or memory-mapped files
        to avoid unbounded RAM usage.
  Complexity: [High] -- Requires disk-backed ring buffer or memory-mapped
  storage, with efficient random access for rendering.

1.3.3  Scrollback Pager (External Viewer)
  What: Pipe the scrollback buffer to an external pager program (less, bat,
        etc.) for browsing.
  Who:  Kit (scrollback_pager config), Foot (pipe-scrollback)
  Complexity: [Medium] -- Serialize buffer to temp file or pipe, launch
  external process.

1.3.4  Scroll-to-Prompt (Shell Integration)
  What: Jump between command prompts in the scrollback using shell
        integration markers (OSC 133).
  Who:  WT (with shell integration), Kit, Wez, iT2 (marks)
  Complexity: [Medium] -- Requires shell integration markers and index
  tracking.

1.3.5  Smooth Scrolling
  What: Pixel-level smooth scrolling animation rather than jumping by whole
        lines.
  Who:  WT, Ala, Kit (partial), Wez, Gho, Con
  Complexity: [Medium] -- Fractional scroll offset tracking; frame
  interpolation in the render loop.


1.4  SEARCH IN TERMINAL OUTPUT
--------------------------------------------------------------------------------

1.4.1  Basic Text Search
  What: Find text in the scrollback buffer and visible area.
  Who:  WT, Kit, Wez, Ala, iT2, Con, Foot, Gho, Rio
  Complexity: [Medium] -- String matching across line-wrapped buffer;
  highlight rendering.

1.4.2  Regex Search
  What: Search using regular expressions for pattern matching.
  Who:  WT, Kit, Wez, Ala (via hints), iT2, Con, Foot
  Complexity: [Medium] -- Integrate regex engine; apply to concatenated
  buffer text.

1.4.3  Case Sensitivity Toggle
  What: Option to toggle case-sensitive vs. case-insensitive search.
  Who:  WT, Kit, Wez, iT2, Con
  Complexity: [Low] -- Flag on the search function.

1.4.4  Highlight All Matches
  What: Visually highlight every occurrence of the search term simultaneously.
  Who:  WT, Kit, Wez, iT2, Con, Gho
  Complexity: [Medium] -- Track all match positions; render overlay or
  modified cell attributes.

1.4.5  Search Direction (Up / Down)
  What: Navigate search results forward or backward through the buffer.
  Who:  WT, Kit, Wez, iT2, Con, Gho
  Complexity: [Low] -- Iterator over match positions with direction flag.

1.4.6  Search Web from Selection
  What: Open a web browser with the selected text as a search query.
  Who:  WT (searchWeb action), iT2
  Complexity: [Low] -- URL-encode selection, launch default browser.


1.5  COPY / PASTE
--------------------------------------------------------------------------------

1.5.1  Standard Copy/Paste
  What: Copy selected text to system clipboard; paste from clipboard.
  Who:  All terminals
  Complexity: [Low] -- Clipboard API integration.

1.5.2  Copy on Select
  What: Automatically copy text to clipboard when mouse selection is made.
  Who:  WT, Kit, Wez, Ala, iT2, Foot, Gho
  Complexity: [Low] -- Hook selection-complete event to clipboard write.

1.5.3  Smart / Word Selection (Double-Click)
  What: Double-click to select a word; configurable word delimiters.
  Who:  WT (wordDelimiters config), Kit, Wez, Ala, iT2, Con, Gho
  Complexity: [Low] -- Character class lookup at click position.

1.5.4  Line Selection (Triple-Click)
  What: Triple-click to select an entire line.
  Who:  WT, Kit, Wez, Ala, iT2, Con, Gho
  Complexity: [Low] -- Select from line start to line end.

1.5.5  Rectangular / Block Selection
  What: Hold Alt+drag to select a rectangular column of text rather than
        line-wrapped selection.
  Who:  WT, Kit, Wez, iT2, Con, Gho
  Complexity: [Medium] -- Column-based selection geometry; handle multi-width
  characters.

1.5.6  Copy with Formatting (HTML/RTF)
  What: Copy selected text preserving colors and font attributes.
  Who:  WT (copyFormatting: html/rtf), iT2, Wez
  Complexity: [Medium] -- Generate HTML/RTF from styled cell data.

1.5.7  Trim Trailing Whitespace on Paste
  What: Automatically remove trailing spaces from pasted text.
  Who:  WT (trimPaste), iT2
  Complexity: [Low] -- String trimming before PTY write.

1.5.8  Large Paste Warning
  What: Prompt user before pasting large amounts of text (e.g., >5 KiB).
  Who:  WT, iT2
  Complexity: [Low] -- Size check + confirmation dialog.

1.5.9  Multi-line Paste Warning
  What: Warn before pasting text containing newlines (prevents accidental
        command execution).
  Who:  WT, iT2, Wez
  Complexity: [Low] -- Newline detection + confirmation dialog.

1.5.10 OSC 52 (Clipboard Access via Escape Sequence)
  What: Allow applications running in the terminal to read/write the system
        clipboard via OSC 52 escape sequences. Commonly used for copying in
        remote SSH sessions.
  Who:  Kit, Wez, Ala, iT2, Foot, Con, Gho
  Note: WT has partial support. Security-sensitive; most terminals restrict
        to write-only or require user confirmation for reads.
  Complexity: [Medium] -- Parse OSC 52; security policy for read vs write;
  base64 encode/decode.

1.5.11 Bracketed Paste Mode
  What: Wrap pasted text in escape sequences (ESC[200~ / ESC[201~) so the
        shell can distinguish pasted text from typed text.
  Who:  All modern terminals
  Complexity: [Low] -- Wrap clipboard content in bracket sequences when mode
  is enabled.


1.6  URL DETECTION AND CLICKABLE LINKS
--------------------------------------------------------------------------------

1.6.1  Auto-Detect URLs
  What: Scan terminal output for URL patterns and underline/highlight them.
  Who:  WT (experimental.detectURLs), Kit, Wez, Ala (hints), iT2, Con, Gho
  Complexity: [Medium] -- Regex-based URL detection across wrapped lines;
  must handle re-detection on reflow.

1.6.2  Ctrl+Click to Open URL
  What: Open detected URL in default browser on Ctrl+click (or Cmd+click).
  Who:  WT, Kit, Wez, Ala, iT2, Con, Gho
  Complexity: [Low] -- Hit test click position against detected URL ranges;
  launch browser.

1.6.3  URL Hints / Keyboard-Driven Link Opening
  What: Assign short labels (a, b, aa, ab...) to all visible URLs for
        keyboard-only opening.
  Who:  Kit (kitten hints), Ala (hints mode), Wez (QuickSelectArgs),
        Foot (urls launch mode)
  Complexity: [Medium] -- Overlay rendering of hint labels; key input capture
  and dispatch.

1.6.4  OSC 8 Hyperlinks (Explicit Hyperlinks)
  What: Support for explicit hyperlinks embedded in terminal output via
        OSC 8 escape sequence. The link text and URL are independent (like
        HTML <a> tags).
  Who:  Kit, Wez, iT2, Con, Foot, Gho, WT (partial/in progress)
  Complexity: [Medium] -- Parse OSC 8; store URL association per cell range;
  render underline and handle clicks.

1.6.5  File Path Detection and Opening
  What: Detect file paths (including line:column references) and open them
        in an editor.
  Who:  Kit (kitten hints --type path), Wez (hyperlink rules), iT2
        (semantic history)
  Complexity: [Medium] -- Pattern matching for paths; editor integration
  configuration.

2. RENDERING AND VISUAL
================================================================================

2.1  GPU-ACCELERATED RENDERING
  What: Use the GPU (via DirectX, Metal, OpenGL, Vulkan, or WebGPU) to
        render terminal text and UI, achieving high frame rates and low
        latency.
  Who:  WT (DirectX/DWrite), Kit (OpenGL), Wez (OpenGL + software fallback),
        Ala (OpenGL), Con (OpenGL), Gho (Metal/OpenGL), Rio (WGPU),
        Foot (Wayland direct, no GPU text rasterization)
  Note: Hyper uses Electron/Chromium compositor (indirect GPU). Foot uses
        CPU rendering but achieves excellent performance on Wayland.
  Complexity: [V.High] -- Requires text atlas management, glyph rasterization
  pipeline, batched draw calls, subpixel positioning, and platform-specific
  graphics API integration. This is typically the largest subsystem in a
  modern terminal.

2.2  FONT LIGATURE SUPPORT
  What: Render programming ligatures (e.g., -> becomes an arrow, != becomes
        a not-equal sign) using OpenType features from fonts like Fira Code,
        JetBrains Mono, Cascadia Code.
  Who:  WT, Kit, Wez, iT2, Con, Gho
  Note: Alacritty explicitly does NOT support ligatures (design philosophy:
        ligatures violate the fixed-grid assumption). Foot does not support
        them either.
  Complexity: [High] -- Requires OpenType shaping engine (HarfBuzz); must
  handle multi-cell glyph runs while maintaining the character grid; reshaping
  on attribute changes.

2.3  EMOJI RENDERING
  What: Display color emoji (Unicode emoji with presentation selectors)
        correctly with proper width (typically 2 cells).
  Who:  WT, Kit, Wez, Ala, iT2, Con, Foot, Gho, Rio
  Complexity: [Medium] -- Requires color font rendering (COLR/CPAL, sbix,
  or CBDT), correct width measurement (wcwidth), and fallback font resolution.

2.4  CUSTOM SHADERS / VISUAL EFFECTS
  What: Apply custom pixel shaders or visual effects to the terminal
        rendering (CRT effect, bloom, film grain, etc.).
  Who:  WT (experimental.pixelShaderPath for HLSL shaders,
        experimental.retroTerminalEffect), Wez (GLSL shaders),
        Kit (custom GLSL shaders), Con (shader support)
  Complexity: [High] -- Requires post-processing pass in the render pipeline;
  shader compilation; passing terminal texture to fragment shader.

2.5  BACKGROUND IMAGE / TRANSPARENCY / BLUR
--------------------------------------------------------------------------------

2.5.1  Background Image
  What: Display a static image behind the terminal text.
  Who:  WT, Kit, Wez, iT2, Con
  Complexity: [Medium] -- Image loading, scaling/alignment, compositing
  behind text layer.

2.5.2  Background Image Animated (GIF)
  What: Support animated GIF as background.
  Who:  WT, iT2, Wez
  Complexity: [Medium] -- GIF frame decoding and timed frame advancement.

2.5.3  Window Transparency (Opacity)
  What: Make the terminal window partially transparent, showing the
        desktop behind.
  Who:  WT (opacity 0-100), Kit (background_opacity), Wez
        (window_background_opacity), Ala (window.opacity), iT2
        (transparency slider), Con, Foot, Gho
  Complexity: [Medium] -- Platform compositor integration (DWM on Windows,
  X11 compositing, Wayland alpha).

2.5.4  Acrylic Material (Windows)
  What: Use the Windows Acrylic translucency effect (blurred, tinted
        background).
  Who:  WT (useAcrylic: true)
  Note: Windows-specific DWM API.
  Complexity: [Medium] -- Windows DWM Acrylic API calls; fallback for
  non-composited desktops.

2.5.5  Mica Material (Windows 11)
  What: Use the Windows 11 Mica material (desktop wallpaper tint effect)
        behind the terminal.
  Who:  WT (via theme.window.mica setting)
  Note: Windows 11 only.
  Complexity: [Medium] -- Windows 11 Mica API integration.

2.5.6  Background Blur (Non-Windows)
  What: Blur the background behind the terminal window on macOS or Linux.
  Who:  Kit (macOS), Gho (macOS blur), iT2 (blur), Hyp (vibrancy on macOS)
  Complexity: [Medium] -- Platform-specific compositor APIs.

2.5.7  Unfocused Window Appearance
  What: Different opacity, background image, or effects when the terminal
        window is not focused.
  Who:  WT (unfocusedAppearance object), Kit (dim unfocused)
  Complexity: [Low] -- Alternate appearance config applied on focus events.

2.6  CURSOR STYLES
  What: Support multiple cursor shapes: block, underline, bar (beam),
        and variants (empty box, vintage/partial block, double underscore).
  Who:  WT (bar, vintage, underscore, filledBox, emptyBox, doubleUnderscore),
        Kit (block, beam, underline), Wez (SteadyBlock, BlinkingBlock, etc.),
        Ala (block, underline, beam), iT2, Con, Foot, Gho
  Complexity: [Low] -- Render different shapes at cursor position.

2.7  CURSOR BLINKING
  What: Configurable cursor blink rate, or disable blinking entirely.
  Who:  All terminals (standard DECSCUSR escape sequences support this)
  Complexity: [Low] -- Timer-driven visibility toggle.

2.8  MINIMUM CONTRAST RATIO ENFORCEMENT
  What: Automatically adjust foreground colors to ensure sufficient contrast
        against the background for accessibility (WCAG compliance).
  Who:  WT (adjustIndistinguishableColors: always/indexed/never),
        Wez (minimum_contrast_ratio), Con
  Complexity: [Medium] -- Color contrast calculation (relative luminance);
  dynamic color adjustment per cell.

2.9  BOLD / ITALIC / UNDERLINE / STRIKETHROUGH TEXT STYLES
  What: Render text with SGR attributes: bold (1), italic (3),
        underline (4), strikethrough (9), dim (2), reverse (7),
        invisible (8).
  Who:  All modern terminals
  Complexity: [Low-Medium] -- Font variant selection (bold/italic faces),
  line drawing for underline/strikethrough.

2.10 INTENSE TEXT STYLE CONTROL
  What: Configure whether SGR bold (1) renders as bold weight, bright
        color, both, or neither.
  Who:  WT (intenseTextStyle: bold/bright/all/none), Kit, Wez, Ala, Gho
  Complexity: [Low] -- Conditional attribute application.

2.11 TRUE COLOR (24-BIT) SUPPORT
  What: Support 16 million colors via SGR sequences
        (ESC[38;2;R;G;Bm for foreground, ESC[48;2;R;G;Bm for background).
  Who:  All modern terminals (WT, Kit, Wez, Ala, iT2, Con, Foot, Gho, Rio,
        Hyp)
  Complexity: [Low] -- Parse SGR 38/48 with color-space parameters; store
  RGB per cell.

2.12 COLORED UNDERLINES (SGR 58/59)
  What: Set underline color independently from text foreground color using
        SGR 58 (set underline color) and SGR 59 (reset underline color).
  Who:  Kit, Wez, Ala, Con, Foot, Gho
  Note: WT has partial support.
  Complexity: [Low] -- Additional color attribute per cell for underline;
  render underline with specified color.

2.13 UNDERLINE STYLES (Curly, Dotted, Dashed, Double)
  What: Support extended underline styles beyond solid: curly/wavy (used by
        LSP diagnostics in vim/neovim), dotted, dashed, double.
  Who:  Kit, Wez, Con, Foot, Gho, Ala (partial)
  Note: WT support is in progress.
  Complexity: [Medium] -- Custom line rendering algorithms for each style
  (bezier curves for curly).

2.14 HYPERLINK SUPPORT (OSC 8)
  What: Render clickable hyperlinks in terminal output where the displayed
        text differs from the URL (like HTML anchor tags).
  Who:  Kit, Wez, iT2, Con, Foot, Gho
  Note: WT support partial/in progress.
  Complexity: [Medium] -- See 1.6.4 above.

2.15 SIXEL GRAPHICS / IMAGE DISPLAY PROTOCOLS

2.15.3 iTerm2 Inline Images Protocol
  What: Display images using the iTerm2 image protocol

  Who:  iT2, Wez, Con
  Complexity: [High] -- Base64 image data parsing; image rendering and
  placement in cell grid.

2.15.4 Image Reflow on Resize
  What: Correctly reposition inline images when the terminal is resized.
  Who:  Kit, Wez (partial), Con (partial)
  Complexity: [V.High] -- Track image placement relative to cell positions;
  recalculate on reflow.

3. SHELL INTEGRATION
================================================================================

3.1  CURRENT WORKING DIRECTORY TRACKING (OSC 7)
  What: The shell reports its current working directory to the terminal
        via OSC 7 escape sequence. Terminal uses this for new tab/pane
        CWD inheritance, tab title display, and drag-and-drop directory
        opening.
  Who:  WT, Kit, Wez, iT2, Con, Gho, Foot
  Complexity: [Low] -- Parse OSC 7; store CWD per pane; use for new
  process startup.

3.2  COMMAND DETECTION / MARKING (OSC 133)
  What: The shell emits OSC 133 markers to denote the start of a prompt,
        the start of user input, the start of command output, and the end
        of command output (with exit code). This enables scroll-to-prompt,
        command output selection, and command history features.
  Who:  WT, Kit (partial), Wez, iT2 (shell integration marks), Gho
  Markers:
    OSC 133;A -- Prompt start (FinalTerm/FTCS_PROMPT)
    OSC 133;B -- Command input start (FTCS_COMMAND_START)
    OSC 133;C -- Command output start (FTCS_COMMAND_EXECUTED)
    OSC 133;D;exitcode -- Command finished (FTCS_COMMAND_FINISHED)
  Complexity: [Medium] -- Parse and index markers; integrate with scroll,
  search, and selection features.

3.3  SEMANTIC ZONES (Prompt, Input, Output Regions)
  What: Using OSC 133 markers, the terminal distinguishes between prompt
        regions, user input regions, and command output regions. This enables
        features like selecting just the output of a command, or right-click
        context menus aware of which zone the user clicked in.
  Who:  iT2 (semantic history), WT (experimental), Wez
  Complexity: [Medium] -- Zone tracking and attribution per cell range;
  expose zones to UI interaction layer.

3.4  COMMAND HISTORY WITH OUTPUT RECALL
  What: Store previous command outputs and allow users to scroll back to
        specific commands, rerun them, or copy their output.
  Who:  iT2 (command history with output), Wez (scrollback search),
        WT (scroll-to-prompt with OSC 133)
  Complexity: [Medium] -- Index of command boundaries with associated output
  ranges; UI for browsing.

3.5  RIGHT-CLICK CONTEXT MENUS WITH SHELL AWARENESS
  What: Context menu options that change based on what was clicked (URL,
        file path, selected text, prompt zone, output zone).
  Who:  WT (experimental.rightClickContextMenu), iT2
  Complexity: [Medium] -- Hit-test click position against zones, URLs, and
  selections; build dynamic menu.

3.6  SHELL-SPECIFIC COMPLETIONS IN TERMINAL
  What: The terminal provides an inline completion/suggestion UI based on
        shell history or integration, separate from the shell's own
        completion system.
  Who:  iT2 (with shell integration), Wez (partial, via leader key)
  Note: Most terminals leave completion to the shell (zsh, fish, etc.).
        This is an advanced integration rarely implemented.
  Complexity: [High] -- Requires deep shell integration, completion protocol,

3.7  PROGRESS BAR / STATUS INDICATION (OSC 9;4)
  What: Show a progress bar in the tab or taskbar based on escape sequences

  Who:  WT (ConEmu-compatible OSC 9;4), Con, Wez

APPENDIX B: IMPLEMENTATION COMPLEXITY GUIDE
================================================================================

The following summarizes the most complex subsystems that a terminal
emulator must implement, ranked by difficulty:

1. GPU Text Rendering Engine [V.High]
   - Text atlas / glyph cache management
   - Subpixel antialiasing
   - Font fallback chain resolution
   - Multi-cell character (CJK, emoji) handling
   - OpenType shaping (HarfBuzz integration)
   - Platform graphics API (DirectX, Metal, OpenGL, Vulkan)

2. VT/ANSI Escape Sequence Parser [High]
   - Hundreds of sequences to support
   - State machine (ground, escape, CSI, OSC, DCS, etc.)
   - Conformance testing against vttest, esctest
   - Edge cases: malformed sequences, embedded data

3. Terminal Buffer / Grid Model [High]
   - Unicode handling (grapheme clusters, combining characters, BiDi)
   - Wide character support (wcwidth)
   - Line wrapping and reflow on resize
   - Scrollback ring buffer with efficient memory usage
   - Attribute storage per cell (colors, styles, hyperlinks)

4. PTY Management [Medium-High]
   - Platform-specific: ConPTY (Windows), posix_openpt (Unix)
   - Process lifecycle management
   - Signal handling (SIGWINCH, SIGHUP)
   - I/O buffering and flow control

5. Image Protocol Implementation [High]
   - Sixel decoder
   - Kitty graphics protocol
   - iTerm2 image protocol
   - Image placement, scrolling, and reflow

6. SSH / Remote Multiplexing [V.High]
   - SSH protocol integration
   - Remote session management
   - State synchronization over network
   - Connection resilience and reconnection

7. IME / Input Method Integration [High]
   - Platform-specific APIs (TSF, IBus, macOS IM)

WINDOWS TERMINAL:
  - Acrylic and Mica material support (Windows compositor integration)
  - JSON fragment extensions for third-party profile/scheme contribution
  - Tab tear-off and cross-window tab drag-and-drop
  - Pixel shader support (HLSL) for custom visual effects
  - Default terminal handler registration (Windows 11)
  - Startup actions for declarative workspace configuration
  - Dynamic SSH profiles from ~/.ssh/config

KITTY:
  - Kitty keyboard protocol (now adopted by many terminals)
  - Kitty graphics protocol (richest image display capability)
  - kitten: framework for writing terminal extensions in Python
  - kitten ssh: transparent remote shell integration
  - kitten diff: terminal-native diff viewer
  - kitten hints: extensible text pattern matching and action framework
  - Remote control over Unix socket

WEZTERM:
  - Lua-based configuration (full programming language for config)
  - Built-in SSH client with multiplexing (no tmux needed on remote)
  - Client-server architecture with detachable sessions
  - Serial port support
  - Comprehensive multiplexer (local and remote domains)
  - Works on Windows, macOS, and Linux with consistent behavior

ALACRITTY:
  - Minimalist philosophy (no tabs, no splits by design)
  - Hints mode for keyboard-driven URL/pattern interaction
  - Vi mode for scrollback navigation
  - Extremely low latency (benchmarked)
  - TOML configuration (clean, human-readable)

ITERM2:
  - tmux -CC integration (tmux control mode rendered as native UI)
  - Triggers (regex-based automatic actions on output)
  - Semantic history (Cmd+click file paths to open in editor)
  - Badges (watermark text in terminal background)
  - Shell integration with command status decorations
  - Password manager integration
  - Instant replay (step through terminal history frame by frame)
  - Annotations on terminal output
  - Coprocesses (pipe terminal I/O to external scripts)

GHOSTTY:
  - Native platform UI (not cross-platform toolkit)
  - Zig implementation for performance
  - Native macOS and GTK Linux integration
  - Focus on correctness and spec compliance

FOOT:
  - Wayland-native (no X11 support, no GPU rendering)
  - Excellent performance through optimized CPU rendering
  - PipeWire/pipesource integration
  - Scrollback search with regex
  - Lightweight and minimal dependency footprint

CONTOUR:
  - VT520-level compatibility ambitions
  - Sixel and Kitty graphics protocol support
  - Colored and styled underlines
  - Extremely spec-compliant VT parser
